# HBSDoc — документирование Handlebars‑компонентов

HBSDoc — это компактная аннотация внутри шаблонов *.hbs* и *.handlebars* по тому же принципу, что JSDoc для JavaScript. Комментарий оборачивается в `{{!-- … --}}`, располагаясь рядом с компонентом.

```hbs
{{!--
    @name Поле ввода

    @parameters                     ← секция параметров

        id: string;

        @description Тип у поля input
        type: string;

        name: string;

        @description Значения для браузерного автозаполнения.
        autocomplete: string;

        placeholder: string;
        value?: string;
        required?: boolean;

        attributes?: {
            name: string;
            value?: string;
        }[]

        withoutRequiredStar?: boolean;
--}}
```

## Теги

| Тег            | Обязательность | Описание                                                                                             |
| -------------- | -------------- | ---------------------------------------------------------------------------------------------------- |
| `@name`        | да             | Человекочитаемое название компонента.                                                                |
| `@parameters` / `@params` | да\* | Секция, внутри которой перечислены **все** входные параметры. Legacy-опечатка `@parametrs` также распознаётся. |
| `@description` | нет            | Описание **следующего** параметра **или** всего компонента, если находится вне секции `@parameters`. |
| `@default`     | нет            | Значение параметра по умолчанию; располагается **до** строки с самим параметром.                     |
| `@type`        | нет            | Определяет alias‑тип, который можно использовать далее в параметрах.                                 |

> `@parameters` / `@params` сами ничего не описывают; это просто маркер начала списка свойств.

### Параметры

* **Синтаксис**: `name[?]: type[,;]`
  `?` — делает параметр необязательным.
* Тип может быть строковым литералом (`"lazy"`), union‑типом (`"eager" | "lazy"`), массивом (`CategoryCardData[]`) или объектом.
* Каждому параметру можно предшествовать блок:

  ```
  @description …
  @default "…"
  paramName: type;
  ```

### Типы (`@type`)

```hbs
{{!--
    …

    @type
    CategoryCardData: {
        name: stringHTML;
        href: stringHref;
        imgSrc: stringSrc;
    }
--}}
```

* После `@type` указывается alias и его тело до пустой строки или конца комментария.
* Alias‑типы резолвятся в hover и signature‑help так же, как встроенные.

## Практические примеры

### 1. Поле ввода

```hbs
{{!-- см. пример в начале --}}
```

### 2. Слайдер с категориями

```hbs
{{!--
    @name Слайдер с категориями

    @parameters

    title: string;                     // string        (обязательный)
    href: stringHref;                  // stringHref    (обязательный)
    items: CategoryCardData[];         // массив alias‑типа
    imgLoading: "eager" | "lazy"      // union‑тип строк

    @type
    CategoryCardData: {
        name: stringHTML;
        href: stringHref;
        imgSrc: stringSrc;
    }
--}}
```

## Рекомендуемые правила

1. **Опциональные** параметры помечайте `?`, а значения по умолчанию — тэгом `@default`.
2. Не пишите описание и параметр в одной строке — для консистентности держите их раздельно.
3. Старайтесь, чтобы описание компонента было **первым** `@description` вне секции `@parameters`.  Остальные `@description` относятся к ближайшему параметру.
4. Если компонент не принимает параметров — не добавляйте `@parameters`.
5. Старайтесь держать alias‑типы (`@type`) в конце блока.
